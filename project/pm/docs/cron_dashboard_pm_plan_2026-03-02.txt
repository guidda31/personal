[PM 계획서] OpenClaw Cron 웹 대시보드 구축
작성일: 2026-03-02
요청자: 구이따
목표: 현재 운용 중인 Cron 목록을 웹에서 한눈에 조회/필터/상태확인 가능한 대시보드 구축

==================================================
1) 범위(Scope)
==================================================
- 대상 데이터: OpenClaw cron list 기준 전체 잡
- 핵심 컬럼:
  - id, name, schedule(expr/tz), next run, last run, status, target, agent
- 요구 기능:
  1. 목록 조회(최신 상태)
  2. 상태 필터(ok/idle/error 등)
  3. 검색(name/id)
  4. 상세 보기(잡 1건)
  5. 수동 새로고침
- 1차는 "조회 중심(Read-only)"으로 제한
  - add/edit/delete/enable/disable는 2차 범위
- 추가요건: 웹페이지로 접속 가능해야 함(브라우저 직접 접속 URL 제공)

==================================================
3) 권장 아키텍처
==================================================
[Collector]
- openclaw cron list 주기 실행(예: 1~5분)
- 결과를 MariaDB(internal_db)에 upsert

[API]
- FastAPI(권장) 또는 Express
- 엔드포인트:
  - GET /api/cron/jobs
  - GET /api/cron/jobs/{id}
  - GET /api/cron/summary

[Web UI]
- 경량 대시보드(서버 렌더 또는 SPA)
- 표 + 필터 + 상태 뱃지 + 마지막 갱신시각

[DB]
- MariaDB internal_db (연결정보는 TOOLS.md 기록본 사용)

==================================================
4) DB 스키마(초안)
==================================================
Table: cron_jobs
- id VARCHAR(64) PK
- name VARCHAR(255)
- schedule_expr VARCHAR(64)
- schedule_tz VARCHAR(64)
- next_run_at DATETIME NULL
- last_run_at DATETIME NULL
- status VARCHAR(32)
- target VARCHAR(64)
- agent VARCHAR(64)
- raw_json JSON NULL
- updated_at DATETIME NOT NULL

Index:
- idx_status(status)
- idx_name(name)
- idx_updated_at(updated_at)

==================================================
5) 구현 단계(Phase)
==================================================
Phase 1 (MVP, 0.5~1일)
- Collector 스크립트 + DB 적재
- 목록 페이지(검색/필터/새로고침)
- 내부망 접속 URL 오픈(예: http://<host>:8088)

Phase 2 (운영형, +0.5일)
- Basic Auth 또는 토큰 인증
- Reverse proxy(Nginx/Caddy) + systemd 서비스화
- 오류 상태 강조/알림 링크
- 자동 새로고침(30~60초)

Phase 3 (확장, +1일)
- 잡 상세(runs 이력 요약)
- Enable/Disable 버튼(권한 제한)
- 텔레그램 보고 연동(일 1회 요약)
- 외부 접속 시 TLS(https) + IP allowlist

==================================================
6) 역할 분담(PM/PL/Planner/Developer/Tester)
==================================================
PM
- 요구사항 고정, 범위 통제, 승인/보고

PL
- 기술 스택 결정(FastAPI + Jinja or React)
- 배포 형태(systemd + nginx/reverse proxy) 확정

Planner
- 태스크 쪼개기(Collector/API/UI/Test)
- 마일스톤 + 리스크 관리

Developer
- collector.py, api.py, dashboard UI, schema.sql 구현

Tester
- 데이터 정합성 검증(원본 cron list vs DB)
- 필터/검색/시간표시 회귀 테스트
- 장애 시나리오(collector 실패/DB 다운)

==================================================
7) 리스크 & 대응
==================================================
- 권한 이슈(openclaw CLI 실행 권한): 서비스 계정 통일
- 시간대 오해(UTC/KST): DB 저장 UTC + 화면 KST 변환
- cron 상태 지연: last_updated 노출로 신뢰성 표시

==================================================
8) 승인 요청(즉시 착수용)
==================================================
아래 4가지 승인 시 바로 구현 착수:
1. MVP는 Read-only로 시작
2. FastAPI + MariaDB 채택
3. 접속 범위: 내부망 우선(외부 공개는 2차)
4. 웹 접속 URL 제공(초기 포트 공개 후 추후 reverse proxy/TLS 전환)
